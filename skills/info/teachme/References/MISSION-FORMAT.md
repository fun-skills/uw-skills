# mission.md 格式

`mission.md` 位于工作区根目录。它记录用户学习该主题的**原因**。所有教学决策——下一步教什么、推荐哪些资源、设计哪些练习——都应当能够回溯到这份文档。

## 模板

```md
# 使命：{主题}

## 为什么
{1–3 句话。用户正在追逐的、具体的现实目标。当他们掌握这项技能后，生活或工作会发生什么改变？避免"为了理解 X"这类抽象表述——往下追问其底层的实际成果。}

## 成功的样子
- {用户将能够做的一件具体、可观察的事}
- {另一件具体的事}
- {……}

## 约束
- {时间、预算、既有承诺、学习偏好，以及任何会限制学习路径的因素}

## 范围之外
- {用户明确表示当前不想追逐的相邻主题——用以保护最近发展区（ZPD）}
```

## 规则

- **一个工作区一个使命。** 如果用户想学两件不相关的事，那就是两个工作区。
- **具体优于抽象。** "在十月前跑完一场半马"胜过"变得更健康"。"给团队交付一个 Rust CLI"胜过"学会 Rust"。
- **对模糊性要追问。** 如果用户说不出"为什么"，先访谈他们再动笔写。**糟糕的使命比没有使命更糟。**
- **现实变了就修订。** 使命会变。当用户的目标移动时，更新这个文件——不要让一份过时的使命继续指挥未来的会话。
- **保持精简。** 如果 `mission.md` 长过一屏，它就已经不再是一个指南针，而变成了一份计划。
